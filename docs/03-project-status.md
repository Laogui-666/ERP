## 3. 技术决策记录

### Decision 1: 全栈框架选型
- **决策**: 使用 Next.js 14 (App Router) 作为全栈框架
- **日期**: 2026-03-19
- **理由**: SSR/RSC/API Routes 一体化，Server Actions 简化表单处理，部署简单，生态成熟

### Decision 2: 数据库选型
- **决策**: 阿里云 RDS MySQL 8.0
- **日期**: 2026-03-19
- **理由**: 甲方指定阿里云 MySQL RDS，托管服务免运维，自动备份，高可用

### Decision 3: 多租户方案
- **决策**: Shared Database + Shared Schema (companyId 字段隔离)
- **日期**: 2026-03-19
- **理由**: 租户数量可控，成本低，维护简单，Prisma 中间件自动注入

### Decision 4: 文件存储
- **决策**: 阿里云 OSS + 预签名直传
- **日期**: 2026-03-19
- **理由**: 文件不经过服务器，减轻带宽，支持大文件，CDN加速

### Decision 5: 短信服务
- **决策**: 暂搁置，仅预留端口和接口定义
- **日期**: 2026-03-19
- **理由**: 甲方明确要求先搁置

### Decision 6: UI 设计风格
- **决策**: 玻璃拟态 (Glassmorphism) + 莫兰迪冷色系
- **日期**: 2026-03-19
- **理由**: 甲方要求"简约高级、冷色调莫兰迪色系、强交互、玻璃拟态"

### Decision 7: 状态管理
- **决策**: Zustand (客户端) + Server Components (服务端数据)
- **日期**: 2026-03-19
- **理由**: 轻量、TypeScript友好、学习成本低

### Decision 8: 角色权限体系
- **决策**: 9级角色，从 SUPER_ADMIN (Lv1) 到 CUSTOMER (Lv9)
- **日期**: 2026-03-19

### Decision 9: 数据库表名隔离
- **决策**: 所有表统一 `erp_` 前缀
- **日期**: 2026-03-20
- **理由**: 同一数据库中有其他项目 14 张表，user/notification 直接冲突，必须隔离

### Decision 10: TypeScript 严格模式
- **决策**: 启用 `exactOptionalPropertyTypes: true`，所有可选字段用 `?? null`
- **日期**: 2026-03-20
- **理由**: 防止 undefined/null 混淆导致运行时错误，已在开发规范中记录

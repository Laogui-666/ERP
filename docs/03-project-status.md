## 3. 技术决策记录

### Decision 1: 全栈框架选型
- **决策**: 使用 Next.js 14 (App Router) 作为全栈框架
- **日期**: 2026-03-19
- **理由**: SSR/RSC/API Routes 一体化，Server Actions 简化表单处理，部署简单，生态成熟
- **备选**: Nuxt.js (Vue 生态，团队若不熟悉 Vue 不推荐)、Express + React (需分别部署，复杂度高)

### Decision 2: 数据库选型
- **决策**: 阿里云 RDS MySQL 8.0
- **日期**: 2026-03-19
- **理由**: 甲方指定阿里云 MySQL RDS，托管服务免运维，自动备份，高可用
- **备选**: PostgreSQL (功能更强，但甲方指定 MySQL)

### Decision 3: 多租户方案
- **决策**: Shared Database + Shared Schema (companyId 字段隔离)
- **日期**: 2026-03-19
- **理由**: 租户数量可控（签证公司），成本低，维护简单，Prisma 中间件自动注入
- **备选**: Schema per Tenant (隔离更好但维护成本高)、Database per Tenant (成本最高)

### Decision 4: 文件存储
- **决策**: 阿里云 OSS + 预签名直传
- **日期**: 2026-03-19
- **理由**: 文件不经过服务器，减轻带宽，支持大文件，CDN加速
- **备选**: 本地存储 (不适合云部署)、AWS S3 (非阿里云生态)

### Decision 5: 短信服务
- **决策**: 暂搁置，仅预留端口和接口定义
- **日期**: 2026-03-19
- **理由**: 甲方明确要求先搁置，后续需要时快速接入阿里云短信服务

### Decision 6: UI 设计风格
- **决策**: 玻璃拟态 (Glassmorphism) + 莫兰迪冷色系
- **日期**: 2026-03-19
- **理由**: 甲方要求"简约高级、冷色调莫兰迪色系、强交互、玻璃拟态"
- **注意**: 需注意性能，backdrop-blur 在低端设备可能有性能问题，需要降级方案

### Decision 7: 状态管理
- **决策**: Zustand (客户端) + Server Components (服务端数据)
- **日期**: 2026-03-19
- **理由**: 轻量、TypeScript友好、学习成本低，配合 Next.js RSC 减少客户端状态需求

### Decision 8: 角色权限体系
- **决策**: 9级角色，从 SUPER_ADMIN (Lv1) 到 CUSTOMER (Lv9)
- **日期**: 2026-03-19
- **理由**: 甲方提供的完整角色体系，覆盖平台运营方和业务使用方

## 5. 风险与待办

### 5.1 已识别风险

| 风险 | 等级 | 影响 | 应对方案 |
|---|:---:|---|---|
| backdrop-blur 性能 | 中 | 低端设备卡顿 | 提供 CSS 降级方案（无模糊的浅色背景） |
| RDS 连接数限制 | 中 | 高并发时连接耗尽 | Prisma 连接池配置 + 连接数监控 |
| 文件上传大小 | 低 | 大文件上传失败 | OSS 直传 + 断点续传 |
| 多租户数据越权 | 高 | 数据泄露 | 底层中间件强制 companyId 过滤 |
| 阿里云服务故障 | 低 | 系统不可用 | 多可用区部署 + 灾备方案 |
